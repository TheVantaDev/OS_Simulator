# Contributing to EduOS 🚀

Welcome, and thank you for considering a contribution to **EduOS**.

EduOS is an interactive, browser-based OS simulator built with modern web technologies. It provides a hands-on learning environment for operating system concepts — CPU scheduling, memory management, process synchronization, and more.

This document explains how to contribute, what to expect, and how your work will be treated throughout the lifetime of the project.

---

## 🧭 Project Direction

EduOS is:

- Open source by default
- Community-driven in spirit
- Designed with a long-term educational mission

For current goals and milestones, see:

- [README.md](README.md)
- [ROADMAP.md](ROADMAP.md)

---

## 🌱 Ways to Contribute

You can contribute in many forms — code is just one of them.

### 🛠 Code

- Bug fixes
- Features aligned with the roadmap
- Performance or architecture improvements
- New educational modules and terminal commands

### 📖 Documentation

- Clarifying existing docs
- Writing guides or explanations
- Improving onboarding for new contributors

### 🎨 Design & UX

- Interface concepts
- Visual language
- Interaction flows

### 🎓 Educational Content

- New learning modules
- Practice exercises and quizzes
- Curriculum design and learning paths

### 🧠 Ideas & Feedback

- Conceptual discussions
- System proposals
- Pedagogical ideas

If you're unsure where your idea fits, open a discussion. Exploration is encouraged.

---

## 🧩 Getting Started

1. Fork the repository
2. Clone your fork locally:
   ```bash
   git clone https://github.com/TheVantaDev/OS_Simulator.git
   ```
3. Create a feature branch:
   ```bash
   git checkout -b feature/your-idea
   ```
4. Commit your changes with clear, meaningful messages
5. Open a Pull Request explaining what you changed and why

Screenshots, videos, or demos are appreciated when relevant.

---

## 🧪 Code & Quality Guidelines

- Follow existing patterns and conventions
- Prefer clarity over cleverness
- Keep changes scoped and intentional
- Document new systems, APIs, or behaviors
- Tests are welcome where applicable, but not mandatory for every PR

Feedback and iteration are part of the process. Requests for changes are about quality, not rejection.

## 🔖 Attribution, Licensing, and Contributor Rights

EduOS is built with transparency and respect for contributors.

### Attribution & Credits

All accepted contributions are permanently acknowledged. Depending on scope and impact, recognition may include:

- GitHub's official contributors listing
- A dedicated [CONTRIBUTORS.md](CONTRIBUTORS.md) file
- Credits in [package.json](package.json) for significant contributions
- An in-application Credits screen

Once credited, attribution is never removed.

- You retain authorship of your contribution and continue to be credited for it.
- Your contribution is licensed under the same open-source license as the project at the time it is accepted.

This agreement allows EduOS to:

- Remain open source
- Distribute builds and binaries

You are not giving up credit, authorship, or recognition. You are allowing the project to grow without legal friction.

## 🔄 Licensing Changes and Transparency

If the project's licensing model evolves in the future:

- Past contributions remain governed by the license in effect at the time they were accepted
